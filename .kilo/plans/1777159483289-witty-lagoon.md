# Plan: Fix GET /api/jobs 500 error

## Problem
`GET /api/jobs` returns HTTP 500.

## Root cause (two-layer failure)
1. **Database schema drift:** The Drizzle `jobsTable` schema includes `researchData: jsonb("research_data")`, but the live PostgreSQL database does **not** have that column. When Drizzle generates the `SELECT` query, it includes `"research_data"`; PostgreSQL rejects the query with `column "research_data" does not exist`, which Drizzle surfaces as a generic "Failed query" error.
2. **Zod validation mismatch (already fixed):** Even if the query succeeded, the generated `ListJobsResponseItem` schema did not allow `researchData`, so Zod `.parse()` would have thrown on any rows that contained it.

## Previously completed
- `researchData` added to OpenAPI `Job` schema in `lib/api-spec/openapi.yaml`.
- Orval code-gen re-run (`lib/api-zod/src/generated/api.ts` now includes `researchData`).
- Type errors in downstream files (`tavily-client.ts`, `jobs.ts`, `jobs/[id].tsx`, `cover-letters/index.tsx`) fixed.
- Full workspace `typecheck` passes cleanly.

## Remaining fix
The DB column still needs to be added.

### Option A: Use `drizzle-kit push` (recommended)
Run:
```powershell
corepack pnpm --filter @workspace/db run push
```
This syncs the schema and adds the missing `research_data` column.

If `push` fails with schema drift:
```powershell
corepack pnpm --filter @workspace/db run compat
```
This runs `runtime-compat.sql` which applies additive migrations.

### Option B: Add `ALTER TABLE` to `runtime-compat.sql`
Insert before any existing `ALTER TABLE jobs` lines (or append at end):
```sql
ALTER TABLE jobs ADD COLUMN IF NOT EXISTS research_data JSONB;
```
Then run `corepack pnpm --filter @workspace/db run compat`.

## Recommendation
Try Option A first. If `push` blocks on drift, fall back to Option B (add to `runtime-compat.sql` and run `compat`).

## Expected outcome
`GET /api/jobs` returns 200 with the jobs list. `GET /api/jobs/:id` works too.
