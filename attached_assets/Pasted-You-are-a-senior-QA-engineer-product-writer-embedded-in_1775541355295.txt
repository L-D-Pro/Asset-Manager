You are a senior QA engineer + product writer embedded in this codebase. Your job is to (1) fully test the current project, (2) determine whether it qualifies as a “test-phase MVP,” and (3) produce a highly detailed user guide for the founder. Then you must ship the guide in two forms: (A) a Markdown file in the repo and (B) a “Guide” page inside the admin/dashboard UI.

GOALS
1) Run a complete end-to-end test pass of the product as it exists today.
2) Decide if the project is currently in “test-phase MVP” status and justify that decision with clear criteria.
3) Create a founder-grade user guide that explains:
   - How to use every feature step-by-step
   - What to test (test plan) and how to test it
   - Common issues and workarounds
   - Settings/configuration guidance (especially AI model/API switching)
   - Known limitations and risk areas
   - Suggested next updates/improvements (prioritized)
4) Add the guide as:
   - docs/USER_GUIDE.md (Markdown)
   - A new dashboard page (e.g., /admin/guide or /dashboard/guide) that renders the same content (or a structured equivalent)

SCOPE OF TESTING
- Treat this as a private internal tool with an admin/back-end focus.
- Test all critical flows:
  - Role profile creation/editing
  - Job ingestion (manual paste, URL import, any existing connectors)
  - Job parsing/extraction and scoring
  - Claims ledger creation/editing and truth-lock behavior
  - Resume/cover letter generation + diff/approval workflow
  - Application tracking/status updates
  - Event logs/audit trails
  - AI model routing/config (OpenRouter): switching models per task, fallbacks, error handling
- If any feature is partially implemented, document what exists vs what is missing.

MVP READINESS CHECK
Define explicit criteria for “test-phase MVP” (minimum viable for private testing), such as:
- Core flows exist end-to-end without manual DB edits
- Errors are surfaced clearly
- Data persists reliably
- Logs/audit trail exist for agent actions
- Basic configuration is possible via admin UI
Then evaluate the current build against those criteria and label it:
- “Test-phase MVP: YES” or “Test-phase MVP: NO”
Include a short rationale and a punch list of blockers if NO.

DELIVERABLE 1: TEST REPORT
Create a concise but complete test report that includes:
- Environment/setup used
- Test cases executed (with pass/fail)
- Bugs found (severity, reproduction steps, expected vs actual)
- Risk areas and edge cases
- Performance/latency notes (especially AI calls)
- Security/config notes (API keys, secrets handling, permissions)

DELIVERABLE 2: USER GUIDE (VERY DETAILED)
Write a founder-facing guide with:
- Quick start (first 30 minutes)
- Full walkthrough of each module/page
- “What to test” checklist per module
- Troubleshooting section:
  - Common errors and what they mean
  - OpenRouter/model errors, timeouts, rate limits, invalid keys
  - Parsing failures, missing fields, dedupe issues
  - Resume diff/approval gotchas
  - Data reset/recovery steps (safe, non-destructive when possible)
- Settings deep dive:
  - Where to set OpenRouter key(s)
  - How to switch models per task
  - How to configure fallbacks and cost-aware routing (if present)
  - Recommended default model choices (if the app supports it)
- “Known limitations” section (honest, explicit)
- “Next improvements” section:
  - Prioritized roadmap (P0/P1/P2)
  - Quick wins vs structural refactors
  - Testing automation suggestions (unit/integration/e2e)

FORMAT REQUIREMENTS
- Markdown file: docs/USER_GUIDE.md
- Dashboard guide page:
  - Add a navigation link labeled “Guide”
  - Render the guide content cleanly (Markdown renderer or structured components)
  - Ensure it works in production build
  - Keep it readable: headings, anchors/table of contents, code snippets where needed

IMPLEMENTATION INSTRUCTIONS
1) Inspect the repo to understand stack, routes, and existing admin UI patterns.
2) Run the app locally, seed minimal data if needed, and execute the test plan.
3) Fix only critical issues that block testing or prevent the guide from being accurate. For non-critical issues, document them.
4) Write the test report and user guide.
5) Add the Markdown file and the in-dashboard Guide page.
6) Verify the Guide page renders correctly and matches the Markdown content (or is clearly equivalent).

OUTPUT
- Commit changes that include:
  - docs/USER_GUIDE.md
  - Guide page route + UI link
  - Any minimal fixes required to complete testing and documentation
- Provide a final summary:
  - MVP status (YES/NO)
  - Top 5 issues found
  - Where the guide lives (file path + dashboard route)
