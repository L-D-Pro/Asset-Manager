Problem Summary
The app’s resume/cover-letter wizard has been unstable around AI generation. The main user-facing failure is: clicking “Generate Resume” often causes long hangs, generic warnings, failed drafts, or diagnostic-only resumes instead of a clean, approvable tailored resume. The terminal shows lots of noisy logs, model fallbacks, JSON/source validation failures, and UI refetch traffic.

The core issue is not one single bug. It is a combination of brittle AI contracts, model routing drift, timeout failures, and UX treating recoverable fallbacks as hard failures.

Key Symptoms

Resume generation used to request strict JSON from models. Multiple models returned malformed/truncated/empty output, especially Kimi/Gemini/Qwen.
OpenRouter showed length, stop, or no visible useful completion, while the app either failed parsing or waited too long.
One Kimi call hung for ~341 seconds and returned empty content:
category: "empty_model_content"
elapsedMs: 341538
Gemini then completed quickly, but returned text that didn’t parse into source-tagged resume lines:
parsedLineCount: 0
validSourceTagCount: 0
Pipeline fell back to deterministic base-resume rendering, but the UI still marked it as “Resume must be regenerated before approval.”
After app reset, base resume was deleted, causing immediate 400s on resume generation until a new base resume was uploaded.
Claim generation also got stuck because claim_generation had no active primary model and fell back into the default chain.
Terminal has lots of repeated GET /claims, /resume-versions, /jobs/:id, /claim-matches, etc. A lot of this is React Query/UI chatter, not paid AI calls.
Important Root Causes

AI contract was too brittle
The original resume pipeline expected nested JSON with source refs. High-reasoning models often failed this when routed through OpenRouter. The app was asking models to act like schema extractors instead of resume writers.

Timeout was not actually enforced
The OpenAI/OpenRouter SDK { timeout } option did not stop at the intended 45s. Kimi ran for 341s. A Promise-level hard timeout was added in:
artifacts/api-server/src/lib/ai-client.ts

Model routing is fragile
resume_tailoring currently resolved to:

Kimi first
Gemini fallback
Config order is DB-driven. If task-specific configs are missing, it falls back to default, which may be slow/expensive/wrong for the task.
Noisy UI refetches obscure the actual issue
Many 304 logs are just wizard/dashboard queries. They should be separated mentally from actual AI calls. Actual AI calls are logged with runId.

Fallback output was valid but treated as invalid
Deterministic fallback could generate a source-supported resume from the base resume, but the UI had a rule that automatically marked deterministic_base_resume_fallback_v1 as needing regeneration.

Reset wipes base resume
The new Admin Reset intentionally clears operational data including base_resume_versions. After reset, resume generation must be blocked until a base resume exists.

Recent Changes Already Made

Added app reset endpoint/page:
POST /api/admin/test-reset
UI: /admin/reset
Added base resume preflight in wizard Step 4.
Claim generation now has:
one AI attempt,
25s timeout,
JSON or line-oriented parsing,
deterministic fallback from pasted notes.
Resume generation moved away from strict JSON toward plain-text/source-tagged resume drafts.
Added hard timeout wrapper in shared AI client.
Reduced resume tailoring timeout to 25s per attempt.
Removed UI rule that automatically rejected deterministic fallback drafts.
