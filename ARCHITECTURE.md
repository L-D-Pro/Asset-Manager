# Architecture

## Monorepo Package Graph

```
pnpm workspace root
│
├── lib/db                        (@workspace/db)
│   └── Drizzle schema + DB client
│       ↑ imported by api-server
│
├── lib/api-spec                  (no package name — script-only)
│   └── openapi.yaml + orval.config.ts
│       → generates lib/api-client-react + lib/api-zod
│
├── lib/api-client-react          (@workspace/api-client-react)
│   └── TanStack Query hooks (generated by Orval from openapi.yaml)
│       ↑ imported by dashboard
│
├── lib/api-zod                   (@workspace/api-zod)
│   └── Zod validation schemas (generated by Orval from openapi.yaml)
│       ↑ imported by api-server (request validation)
│
├── lib/integrations/integrations-openrouter-ai  (@workspace/integrations-openrouter-ai)
│   └── OpenRouter client (OpenAI SDK-compatible, Replit-managed credentials)
│       ↑ imported by api-server pipelines
│
├── artifacts/api-server          (@workspace/api-server)
│   └── Express 5 REST API — reads lib/db, lib/api-zod, integrations-openrouter-ai
│
└── artifacts/dashboard           (@workspace/dashboard)
    └── React + Vite SPA — reads lib/api-client-react
```

TypeScript project references are declared in the root `tsconfig.json` for `lib/db`, `lib/api-client-react`, `lib/api-zod`, and `lib/integrations-openrouter-ai`. Artifacts (`artifacts/*`) have their own `tsconfig.json` files that extend `tsconfig.base.json`.

## Request Lifecycle (HTTP → DB)

```
Browser/Client
  │  HTTP request to /api/*
  ▼
Express 5 router (artifacts/api-server/src/app.ts)
  │  Route matched → handler in artifacts/api-server/src/routes/<entity>.ts
  ▼
Zod validation (lib/api-zod generated schemas)
  │  Request body / query params validated; 400 returned on failure
  ▼
Business logic / service function
  │  Drizzle ORM queries against PostgreSQL
  ▼
lib/db (@workspace/db)
  │  Returns typed rows
  ▼
Handler formats response → JSON → Browser
```

For AI-triggering routes (parse, tailor, cover letter), the handler calls into the pipeline layer:

```
Route handler
  │  Calls pipeline function (artifacts/api-server/src/lib/pipelines/*.ts)
  ▼
Pipeline
  │  Calls callAI() (artifacts/api-server/src/lib/ai-client.ts)
  ▼
AI Client
  │  selectModelForTask() resolves active model from ai_model_configs
  │  Calls OpenRouter via @workspace/integrations-openrouter-ai
  │  Logs every attempt (success + failure) to event_logs
  ▼
Pipeline
  │  Parses JSON response
  │  Runs truth-lock validation (validation.ts)
  │  Persists result as ResumeVersion / CoverLetterVersion in status=pending_approval
  ▼
Route handler returns the created record
```

## AI Pipeline Data Flow

```
1. User triggers "Parse JD" on a job
   → POST /api/jobs/:id/parse
   → jd-parse.ts sends rawJdText to AI (jd_parsing task scope)
   → AI returns structured JSON (skills, responsibilities, seniority, salary, remote type)
   → Job record updated with parsedRequiredSkills, parsedKeywords, etc.

2. User triggers "Tailor Resume"
   → POST /api/jobs/:id/tailor
   → resume-tailor.ts: matchClaimsToJob() ranks Claims Ledger against parsed JD
   → Top 15 claims sent to AI (resume_tailoring task scope)
   → AI returns tailored bullets with claimIds attributing each bullet to claims
   → validateBullet() drops any bullet whose claimIds don't exist in selectedClaims
   → assertMinimumContent() throws TruthLockViolation if zero valid bullets remain
   → New ResumeVersion created in pending_approval state with diffData

3. User reviews and approves/rejects each change in the dashboard
   → POST /api/resume-versions/:id/approve (state machine: pending_approval → approved)
   → POST /api/resume-versions/:id/reject  (state machine: pending_approval → rejected)
   → Any other initial status returns 409 Conflict

4. Cover letter generation follows the same pattern (cover-letter-draft.ts)
   → POST /api/jobs/:id/cover-letter
   → AI returns annotated paragraphs with claimIds
   → validateParagraph() enforces claim attribution on body/hook paragraphs
   → CoverLetterVersion created in pending_approval state
```

## Key Design Decisions

### Spec-First API (OpenAPI → Orval → generated clients)
The OpenAPI spec (`lib/api-spec/openapi.yaml`) is the single source of truth. Orval generates two artifacts from it:
- `lib/api-client-react`: TanStack Query hooks for the React dashboard
- `lib/api-zod`: Zod schemas used by the API server for request validation

This means adding an endpoint requires editing `openapi.yaml`, running `pnpm --filter @workspace/api-spec run codegen`, and then wiring the handler. Type safety flows from spec to both client and server.

### Claims Ledger as Truth Lock
Resume bullets can only be generated from approved Claims in the Claims Ledger. The AI is given a list of selected claims and instructed to attribute every bullet to one or more claim IDs. The `validateBullet()` function discards any bullet that references a claim ID outside the provided set. This makes hallucination structurally impossible — not just a prompt instruction.

### Human-in-the-Loop State Machine
All AI outputs are persisted in `pending_approval` status. The backend enforces state machine transitions:
- `pending_approval` → `approved` or `rejected` (via the approve/reject endpoints)
- Any other transition returns 409 Conflict

This prevents AI content from ever being used without explicit human review.

### Cost-Aware Model Routing
The `ai_model_configs` table allows multiple model configs per task scope, with `priority` (lower = tried first) and `fallbackModelId` chain. Every AI call logs token counts and estimated USD cost to `event_logs`, enabling cost monitoring and auditability.

### EventLog as Immutable Audit Log
All state changes, AI calls (success and failure), and key operations are written to `event_logs`. These records are append-only and read-only via the API (no update/delete endpoints). This is the system's audit trail.

### Separate Workspace Packages for Generated Code
Generated code (`api-client-react`, `api-zod`) lives in separate workspace packages rather than being inlined. This avoids import cycles, enables independent caching, and makes the generation boundary explicit.
